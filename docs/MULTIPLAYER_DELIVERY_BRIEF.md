# 多人功能交付约定

更新日期：2026-07-22。

本文记录用户确认的多人版本完整交付要求，是 `feat/multiplayer` 开发期间的范围基线和最终验收
清单。技术实施细节见 [`MULTIPLAYER_PLAN.md`](MULTIPLAYER_PLAN.md)。若实现选择与本文冲突，必须先
向用户说明并获得确认，不能自行缩减产品范围。

## 1. 总目标

完成一个可以实际游玩的 2–4 人联机版本，而不是只搭建后端房间骨架或演示页面。交付范围同时
包含：

- 前端完整产品流程；
- 后端共享房间运行时；
- 用户、Session、成员、邀请和调查员绑定；
- 多人规则、权限、事件隔离和断线恢复；
- Electron 单机/联机双模式；
- 浏览器联机模式；
- 数据库迁移、自动化测试、Azure 部署和文档。

任何阶段都不能以“后端接口已经存在”代替可操作的用户界面，也不能以“前端按钮已经出现”代替
真实、安全、可恢复的多人后端。

## 2. 不可妥协的产品要求

### 2.1 保留 Electron

Electron 是正式支持的客户端，不能因多人功能改成纯网页项目。Electron 启动后必须提供清晰的
模式选择：

```text
启动 Electron
├─ 单机游戏
│  ├─ 连接/启动本地 FastAPI
│  ├─ 使用本地 SQLite
│  ├─ 默认不要求账号登录
│  └─ 保留现有开局、存档和游戏体验
└─ 多人游戏
   ├─ 连接 Azure HTTPS/WSS
   ├─ 登录或注册
   ├─ 进入联机大厅与房间流程
   └─ 使用服务器 PostgreSQL 和权威房间运行时
```

两种模式复用游戏界面、消息渲染、角色面板和协议类型，但连接地址、认证状态、房间状态和后端
生命周期必须明确隔离。切换模式不能串用 Cookie、世界、存档或 WebSocket 状态。

### 2.2 联机前端必须完整

联机相关界面要根据实际产品流程补齐，不局限于最初举例。最低要求包括：

1. **模式选择页**
   - 单机游戏按钮；
   - 多人游戏按钮；
   - 当前模式和服务器连接状态。

2. **认证流程**
   - 登录、注册、退出；
   - Session 恢复与过期处理；
   - 表单校验、加载、错误和限流反馈；
   - 云端未登录时不连接游戏 WebSocket。

3. **联机大厅**
   - 我的房间/可进入世界列表；
   - 创建房间；
   - 输入邀请码或房间码加入；
   - 房间人数、状态、模组、本人角色和最后更新时间；
   - 空列表、加载失败、邀请过期和权限不足状态。

4. **创建房间流程**
   - 选择模组和基础房间设置；
   - 创建中的明确反馈；
   - 创建成功后直接进入房间；
   - 重复提交和创建失败的恢复。

5. **房间页面**
   - 房间名称、模组、房主和状态；
   - 成员列表、在线/离线、房主/玩家/旁观者标识；
   - 调查员选择、占用、释放和冲突提示；
   - 准备/取消准备；
   - 邀请链接或房间码的创建、复制和撤销；
   - 房主修改权限、移除成员、移交房主和开始游戏；
   - 普通成员退出房间；
   - 浏览器刷新或 Electron 重启后重新进入原房间。

6. **多人游戏界面**
   - 当前行动者的显著提示；
   - 非当前行动者的只读/等待状态；
   - 在线成员、调查员和掉线状态；
   - 当前行动者提交中、排队、拒绝和完成反馈；
   - 房主跳过掉线玩家、转交行动权和必要管理操作；
   - 私人线索、私人决定与公共叙事的清晰区分；
   - 断线、重连、补发和完整恢复状态；
   - 返回房间、退出房间和关闭窗口时的明确行为。

7. **全局交互质量**
   - 所有异步操作具有 loading/success/error/empty/disabled 状态；
   - 不允许无反馈按钮、静默失败或只能看控制台排错；
   - 权限不足时隐藏危险操作并由后端再次拒绝；
   - 房主、玩家、旁观者和当前行动者在视觉上易于区分；
   - 保持现有克苏鲁视觉语言，同时满足常用桌面尺寸和浏览器响应式；
   - 键盘操作、焦点、对话框和基本无障碍语义不能明显退化。

前端可以为完整交互新增页面、路由、弹窗、侧栏、状态条、通知和设置项，不受现有组件数量限制。

## 3. 后端与安全要求

联机采用 Azure 权威服务器，不做 P2P。必须保证：

- 模型 API Key、模组秘密、NPC 秘密和工具参数只存在服务端；
- 一个活跃 `world_id` 只有一个共享 `GameRoom` 和一个 `GameEngine`；
- 多连接共享同一模型消息历史，而不是各自创建引擎；
- 行动由服务端当前行动者规则和串行队列裁决；
- `action_id` 幂等，重复点击不会重复调用模型、工具或提交数据库；
- 用户身份来自 Session，调查员身份来自服务端绑定，不信任客户端 `user_id/actor_id`；
- 公共、指定玩家、房主和 `server_only` 事件在发送前完成过滤；
- 房间事件具有单调 ID、确认、增量补发和完整恢复路径；
- 世界状态、回合、事件、快照和自动存档保持事务一致；
- 不同房间之间的引擎、事件、消息历史、成员和状态不得串线。

## 4. 分工与协作方式

### Codex（主执行者）

- 负责后端、数据库、Alembic、房间运行时、权限、协议和 Azure 部署；
- 先定义前后端共享的 HTTP/WS 类型、错误码和事件时序；
- 为 Kimi 提供稳定契约、mock/fixture 和联调说明；
- 审核 Kimi 的前端改动，运行完整测试并负责最终集成；
- 维护计划、架构、API、数据库、部署和用户文档；
- 发现跨边界问题时主动与 Kimi 对齐，不等待最后一次性拼接。

### Kimi（前端执行者）

- 通过名为 `kimi` 的 tmux 会话协作；
- 负责 Electron/浏览器的模式选择、认证、大厅、房间和多人游戏交互；
- 负责前端状态管理、协议消费、错误/等待/恢复状态和视觉实现；
- 可以在产品范围内新增必要页面与组件；
- 不自行假设服务端权限或私密事件语义，协议不明确时与 Codex 对齐；
- 提交前端测试并配合真实双客户端联调。

双方共享同一工作区，因此必须在动手前划分文件范围，避免覆盖对方未提交改动。Codex 负责最终
代码审查、冲突处理和分阶段提交。

## 5. 开发与部署约束

- 全部工作先在 `feat/multiplayer`，未通过验收不得直接合入 `master`；
- 前端和后端按可回滚的阶段提交，不堆积一个超大提交；
- 数据库变化必须有 Alembic、SQLite 测试和真实 PostgreSQL 验证；
- 协议变化同步更新 `docs/API.md`，运行时变化同步更新 `docs/ARCHITECTURE.md`；
- 单机模式回归测试必须持续通过；
- Azure 先部署隔离 staging，不用未完成版本直接替换生产；
- 旧版 Azure 服务可以停止，但 Nginx、PostgreSQL、备份和现有数据不得误删；
- 不修改用户 SSH 密码或凭据；生产秘密不得写入仓库、日志、测试 fixture 或聊天转交内容。

## 6. 完成定义

只有以下条件全部满足，才能称为“多人路线完成”：

- Electron 中能选择单机或多人，单机原有功能正常；
- 两个新账号能注册/登录、创建/加入房间、选择不同调查员并准备；
- 房主能开始游戏并管理成员和行动顺序；
- 两台客户端看到一致公共叙事，只有当前行动者能提交；
- 检定、SAN、HP、物品、战斗和私人决定作用于正确调查员；
- 私人事件不会出现在另一位玩家的网络消息、日志或恢复数据中；
- 并发点击、重复提交、掉线、刷新、重连和服务重启不会重复回合或损坏状态；
- 存档、读档和时间线恢复在多人模式下行为明确并通过测试；
- 浏览器联机版和 Electron 联机版均通过实际双客户端 E2E；
- Azure staging 通过 TLS/WSS、权限、备份、恢复、重启和回滚演练；
- 完整 Python、前端、构建、迁移和 PostgreSQL 集成测试通过；
- 用户文档能够指导非开发者完成登录、创建房间、邀请和开始游戏。

技术骨架、静态页面、仅单客户端演示或只能在开发机运行，均不满足完成定义。

## 7. 执行前检查

正式开始每个阶段前，Codex 必须确认：

- 本阶段对应本文哪一项交付要求；
- 前后端契约是否已确定；
- Codex 与 Kimi 的文件范围是否冲突；
- 单机兼容、权限、可见性、并发和失败恢复如何测试；
- 是否需要数据库迁移、API/架构文档和 Azure staging 变更。

阶段结束时按同一组问题复核，不能只报告“代码已写完”。
