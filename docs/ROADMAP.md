# TRPG Master 开发路线图

更新日期：2026-07-22。本文件描述当前仓库之后的工作，不再把已经完成的文件存储、数据库或
账号系统列为未来计划。

## 1. 当前基线

已经完成：

- 一个 `GameEngine`、一个 `ModelSession` 和一份连接内消息历史；
- 一条 LangGraph 回合工作流，在普通叙事职责节点和战斗职责节点间路由；
- 确定性检定、战斗、SAN、物品、发现规则和素材事件；
- SQLAlchemy 2 + Alembic；桌面 SQLite、生产 PostgreSQL；
- `world_states` JSONB、关系型回合/事件、不可变快照、存档、玩家笔记和模型用量；
- Argon2id 用户、可撤销服务端 Session、世界成员角色、审计、HTTP/WS 权限；
- 旧 `worlds/` 的一次性幂等导入和生产加密备份配置；
- 按 `world_id` 隔离、revision 并发控制、时间线分支和断线恢复。

当前不是多 Agent 系统。代码中的 `call_story_agent` 和 `call_combat_agent` 是历史节点名：两者
共享同一引擎、消息、状态和模型调用器。战斗节点只是在本轮切换模型并叠加临时职责提示。

当前也不是共享多人房间。账号和世界成员权限已经存在，但每条 WebSocket 仍创建自己的
`GameEngine` 和消息历史；同一世界尚无共享行动队列和事件广播。

## 2. 近期优先级

### M1：数据库上线与运维收口

- [ ] 在 Azure VM 创建 `trpg_app` 和 `trpg_master`，仅监听 `127.0.0.1`；
- [ ] 在维护窗口执行 Alembic 和旧世界导入，核对世界、回合、存档、分支与笔记数量；
- [ ] 启用 `TRPG_REQUIRE_AUTH=1`、关闭公开注册并配置可信 Origin；
- [ ] 启用每日加密备份 timer，并完成一次异机恢复演练；
- [ ] 记录数据库连接池、磁盘、备份大小和迁移耗时；
- [ ] 为部署失败补充回滚与操作手册。

验收标准：重启和发布不丢数据；数据库与旧目录恢复结果一致；未授权用户无法读取或切换他人世界。

### M2：共享房间运行时

- [ ] 实现 `RoomManager` / `GameRoom`，一个活跃房间只持有一个共享 `GameEngine`；
- [ ] 把房间成员映射到现有 `users/world_members`，不信任客户端 `actor_id`；
- [ ] 建立串行行动队列、公平性规则、暂停和空房休眠；
- [ ] 增加房间级 `event_id`、逐连接 ack 和增量补发；
- [ ] 区分公共、指定玩家和 GM 私有事件；
- [ ] 增加双连接并发、重连、进程重启和跨房间隔离测试。

验收标准：两个客户端加入同一房间时只产生一条 GM 历史；并发动作严格串行；重连不会创建第二个引擎。

### M3：双调查员规则与 UI

- [ ] 将单数 `world_state.pc` 演进为调查员集合，并提供旧存档迁移；
- [ ] 为检定、SAN、物品和战斗操作显式传入服务端授权的角色 ID；
- [ ] 实现角色占用、当前提交者和战斗行动者 UI；
- [ ] 把私人决定、私人线索和掉线托管发给正确成员；
- [ ] 完成两名玩家的探索、检定、线索、战斗、存档和恢复 E2E。

### M4：多人体验与可观测性

- [ ] 房主移交、踢出、旁观、房间归档和复制；
- [ ] 行动提案/投票与关键决定确认；
- [ ] 房间延迟、队列、模型 Token、错误率与成本报表；
- [ ] 2–4 人长时间压力测试和故障注入；
- [ ] 大型冷快照迁移对象存储，数据库保留索引、哈希和权限。

## 3. 关于 Agent 的后续决策

近期不拆分第二个独立 Agent。优先使用：

- 确定性领域服务；
- 同一工作流中的职责节点；
- 结构化状态和数据库事务；
- 有触发条件的 Skill/Lorebook 注入。

只有固定评测证明单工作流在某一类任务持续失败，而且问题不能通过状态、工具或提示边界解决时，
才评估独立 Agent。候选必须满足：

- 有明确、稀疏的触发条件，而不是每回合串行调用；
- 有结构化输入输出和延迟/Token 预算；
- 不持有第二份世界事实；
- 不绕过权限、骰子、战斗或数据库事务；
- 有叙事一致性、工具正确率、信息泄露和恢复测试。

不计划采用每个 NPC 一个 Agent，也不计划串行调用“战术 Agent → 规则 Agent → 叙事 Agent”。

## 4. 持续质量门槛

- 完整 Python 测试、Ruff、迁移检查和真实 PostgreSQL 集成验证；
- 桌面启动、浏览器、Electron 和旧世界导入冒烟测试；
- API/数据库 schema 变化同步更新 `docs/API.md`、`docs/ARCHITECTURE.md` 和 Alembic；
- 每次发布验证备份可读，定期执行恢复而不只检查备份文件存在；
- 新增多人状态前先定义所有权、可见性、并发和失败恢复语义。
