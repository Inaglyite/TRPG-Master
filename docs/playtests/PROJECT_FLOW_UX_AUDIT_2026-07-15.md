## 总体体验

正常网络下，当前基础流程是可玩的：角色状态在开场前同步；模型首段到达后是细粒度流式输出；点击选项后立即禁用输入；确定性侦查约 1 秒内展示；工具调用和后续叙事顺序正常；最终选项能从叙事中解析为按钮。

这一轮的实际延迟如下：

| 环节 | 实测 |
| --- | ---: |
| 开场首 token | 13.80 秒 |
| 开场完成 | 37.76 秒 |
| 调查动作到侦查结果 | 约 1 秒 |
| 调查回合第一次模型调用 | 15.42 秒（首 token 12.39 秒，2 个工具） |
| 工具后续写 | 26.39 秒（首 token 8.86 秒） |

修复前的最大问题不是正常路径，而是回合中断后的事实一致性和恢复能力：断线可能让服务端世界已经推进、浏览器却只看到半句故事。该问题现由持久回合提交与重放覆盖。

## P0：断线后服务端事实与玩家可见叙事分叉，重连界面永久卡死

### 复现步骤

1. 开始新游戏并等待开场完成。
2. 点击“在大厅中仔细检查地毯上的污渍和周围的陈设”。
3. 前端先显示侦查 `d100=9 vs 38`，困难成功。
4. 在后续叙事流式输出到“那是很久以前的事了——不小心”时停止后端。
5. 后端模型线程继续执行：日志显示 `state_add_clue`、`npc_reveal`，随后完成第二次模型请求。
6. 重启后端，等待浏览器自动重连。

### 实际结果

- 页面保留骰点和半句叙事，没有收到该回合的 `done`，也没有行动选项。
- 重连成功后输入框仍为 disabled，placeholder 仍是“守秘人正在叙述……”。
- 新连接只请求 `state`，不会查询未完成回合、重放事件或恢复自动存档叙事。
- 服务端的线索/NPC 事实已经被工具提交；最终叙事也已在后台完成并自动保存，但当前页面不知道。
- 玩家只能刷新/重新读档；界面没有说明“本轮结果可能已提交”，也没有恢复按钮。

### 原因

- 前端仅在 `done` 中调用 `onDone()` 恢复输入；`onclose` 只清理本地 turn 标识。
- 重连后只发送 `ping` 和 `state`，没有会话/回合恢复协议。
- 工具会在模型续写前立即写入世界；`finalize` 才保存消息并发送 `done`。因此断线窗口内天然可能出现“事实已写、可见叙事未交付”。
- 每个 WebSocket 重连都会创建新的 `GameEngine`，旧连接的内存消息和事件队列无法附着到新连接。

### 建议验收标准

- 最低限度：断线时结束 loading、关闭失效弹窗并显示明确恢复入口；重连后允许“从最近自动存档恢复”，不能保持假性生成状态。
- 完整方案：服务端持有 world/session 级回合记录，保存 `turn_id`、状态、最终叙事和可重放事件；客户端用 last acknowledged seq 重连。
- 工具事务与叙事交付需定义一致性边界：要么在回合记录中持久化已提交但未展示的事件，要么支持从权威回合结果重建 UI。

## P1：重连失败会每 5 秒追加一条相同错误消息

停止后端后，浏览器每次自动重连失败都会执行新的 `onclose`，实际页面连续出现三条“连接断开。5秒后重试……”，继续离线会无限增长。

建议只维护一个可更新的连接状态条/气泡，并采用有上限的指数退避；成功重连后将同一条状态更新为“已重连”，不要再追加一条“已连接”。

## P1：同一连接可以排队多个回合，但事件流只能表示一个 active turn

服务端对每个 `action` 先调用 `begin_turn()`，之后才把工作放入由 `turn_lock` 串行的 executor。若同一连接在前一轮运行时又发送 action（脚本、异常客户端、重连发送队列或未来 UI 回归均可触发），第二次 `begin_turn()` 会立即覆盖事件流的 active turn id，而第一轮还在产生回调。

结果是第一轮后续事件可能被标成第二轮，两个 worker 虽串行执行，协议生命周期却已经交叠。前端会接受错误归属的事件，随后可能丢弃真正第二轮事件或提前执行 `done`。

建议在进入 `begin_turn()` 前非阻塞获取“回合已占用”状态，忙时明确拒绝/排队 action；只有 worker 真正开始时才创建 turn。新增测试应覆盖“第一轮流式输出期间连续发送两个 action”。

## P1：角色选择中同一角色重复出现，来源差异不够可见

疯狂宅邸角色页实际显示三个卡片：默认霍华德、默认黄千陆、模组特色霍华德。两个霍华德的卡片主信息完全相同，只有所在分组不同；滚动或窄窗口中很容易误以为重复渲染或选错来源。

原因是角色服务不做跨来源去重，卡片本身也不展示 `source_label`。旧《猩红文档》报告里的同名黄千陆问题仍属于同一缺陷。

建议给卡片加明确来源徽标，并在内容/ID相同的情况下合并或解释覆盖关系；至少让确认区显示“霍华德 · 模组特色/默认调查员”。

## P1：开场替玩家做了有意义的行动和对白

本次开场中，系统在玩家第一次获得操作权前自动完成了：停车、走进宅邸、让门在身后锁死、向管家取出匿名信并说“我是收到一封信才来的”。这不是单纯布景，而是替玩家决定了进入危险地点、展示证物和对白内容。

这会削弱 TRPG 最重要的行动自主感，也可能与调查员性格冲突。建议开场只推进到第一个真正选择点；若模组必须以“已经进入宅邸”为硬前提，应在模组公开前提中明确，并禁止模型继续代写玩家对白或主动展示物品。

## P2：行动选项仍依赖从自然语言/Markdown 反向解析

`done` 后前端从最后一条 GM HTML 中寻找“你可以”及最后一个 `<ol>`；找不到标记时甚至会直接采用最后一个有序列表。这会把叙事中的编号证据、步骤或名单误当作行动按钮。模型格式稍变也会退回自由输入，已有旧报告“一回合说你可以但没有选项”的表现。

建议让服务端在 finalize 后发送结构化 `choices`；正文裁掉选项菜单，前端不再从展示 HTML 猜协议数据。自然语言解析只保留为旧存档/第三方兼容降级。

## P2：等待体验缺少阶段反馈，复杂工具回合显得像停住

开场 13.8 秒才出现首字；调查动作先在 1 秒内出现骰点，但之后约 12 秒才出现模型反馈。工具调用后又等待约 8.9 秒才续写。UI 全程主要只有“守秘人正在叙述”，玩家无法区分：排队、检定结算、模型思考、工具执行、二次续写或连接异常。

建议提供少量稳定阶段文案（“正在检定”“正在整理结果”“正在续写场景”）和超过阈值后的耗时提示；不要暴露内部工具名或思维链。

## P2：建议检定与自动预检的交互契约不一致

架构/API 仍把 `suggest_check -> suggest_reply` 描述为常规阻塞握手，但本次点击明确调查选项后直接掷骰，没有询问玩家是否确认。两种体验都可以成立，但当前产品语义不清：选择“仔细检查”是否已视为确认检定？自由文本调查又何时弹确认？

建议明确规则：按钮行动若已清楚标注会检定，则点击即确认；自由文本触发额外检定时才弹窗。界面选项可带“侦查”标签，减少突兀骰点。

## P3：浏览器请求 `/favicon.ico` 返回 404

不影响玩法，但每次打开页面都会产生一条无意义 404 日志。补一个 favicon 或显式空图标即可。

## 已确认正常的部分

- 当前 FIFO 事件流在单一正常回合内保持了骰点、工具调用、续写、`done` 的顺序；相关单元测试也通过。
- 明确调查动作的权威预检生效，本轮真实使用角色侦查值 38，并展示结构化 d100 动画数据。
- 模型确实调用了 `state_add_clue` 与 `npc_reveal`，不是只在文本中伪造发现。
- 工具后的模型续写能看到权威结果，管家的反应与已提交 NPC 揭示一致。
- 自动槽在正常 finalize 时更新；167 项现有测试全部通过。
- 前端生产构建无 TypeScript 错误。

## 建议修复顺序

1. 先修断线/重连与未完成回合恢复；这是数据事实和玩家认知分叉的问题。
2. 修同连接多 action 的 turn 生命周期竞争，并补协议级并发测试。
3. 将选项改为结构化事件，顺便改善等待阶段反馈。
4. 收紧开场玩家代理边界，并解决同名角色来源辨识。
5. 最后处理检定确认语义和 favicon 等低风险体验项。
